Authorization in Qeet ID is **RBAC-first** with a single-call check, **group-level
roles**, an **explainable** mode that returns the grant-path trace, and an **ABAC
policy** layer for rules RBAC can't express.

## Permissions & roles

A **permission** is a string like `billing:write`. A **role** bundles permissions and
is assigned to a **user** or a **group** within a tenant. A user's **effective
permissions** are `direct ∪ group-derived`.

<APIEndpoint method="GET" path="/v1/permissions" title="List permissions" />
<APIEndpoint method="POST" path="/v1/tenants/{tenantID}/roles" title="Create a role" />
<APIEndpoint method="PUT" path="/v1/roles/{roleID}/permissions/{permID}" title="Add a permission to a role" />

## Assigning roles

To a **user**:

<APIEndpoint method="PUT" path="/v1/users/{userID}/tenants/{tenantID}/roles/{roleID}" title="Assign a role to a user" />

To a **group** (every member inherits the role's permissions):

<APIEndpoint method="PUT" path="/v1/tenants/{tenantID}/groups/{groupID}/roles/{roleID}" title="Grant a role to a group" />
<APIEndpoint method="GET" path="/v1/tenants/{tenantID}/groups/{groupID}/roles" title="List a group's roles" />

Inspect a user's combined effective permissions in a tenant:

<APIEndpoint method="GET" path="/v1/users/{userID}/tenants/{tenantID}/permissions" title="Effective permissions" />

## The check — single call

The hot-path question "may this principal do X?" is one request:

<APIEndpoint method="GET" path="/v1/check" title="Authorize an action" />

```bash
curl "https://api.id.qeet.in/v1/check?user_id=$U&tenant_id=$T&permission=billing:write" \
  -H "Authorization: ApiKey $QEETID_API_KEY"
# → { "allowed": true }
```

<Tabs items={['TypeScript', 'Go', 'Python']}>

```ts tab="TypeScript"
const ok = await qeetid.can({ user: userId, tenant: tenantId, permission: "billing:write" });
// canAll(user, tenant, ["a", "b"]) → true only if all pass
```

```go tab="Go"
ok, _ := qeetid.Can(ctx, qeetidsdk.PermissionCheck{User: userID, Tenant: tenantID, Permission: "billing:write"})
```

```python tab="Python"
ok = qeetid.can(user=user_id, tenant=tenant_id, permission="billing:write")
```

</Tabs>

## Group-level roles

Assigning a role to a **group** grants it to every member transitively — the basis for
directory-driven access. SCIM keeps group membership in sync (see
[Enterprise → SCIM](/docs/enterprise/scim)), so adding someone to a group in Okta/Entra
grants their roles in Qeet ID automatically.

## Explainable authz

Add `?explain=true` to `/check` and instead of the bare `{ allowed }`, Qeet ID returns
the full **grant-path trace**: every distinct grant (direct and group-derived) that
confers the permission, or — on a denial — the reason. Most CIAM platforms can't tell
you *why* a decision was made; Qeet ID can.

<APIEndpoint method="GET" path="/v1/check?explain=true" title="Authorize + explain" />

```bash
curl "https://api.id.qeet.in/v1/check?user_id=$U&tenant_id=$T&permission=billing:write&explain=true" \
  -H "Authorization: ApiKey $QEETID_API_KEY"
```

```json title="AuthzExplanation (allowed)"
{
  "allowed": true,
  "paths": [
    {
      "permission": "billing:write",
      "granted_by": "role:billing-admin",
      "via": "group:finance-team",
      "group_id": "…",
      "role_id": "…"
    }
  ]
}
```

```json title="AuthzExplanation (denied)"
{
  "allowed": false,
  "paths": [],
  "reason": "no role grants billing:write for this user in this tenant"
}
```

<Callout type="tip" title="It's in the dashboard too">
  The admin dashboard's <strong>Access Tester</strong> renders this grant-path trace
  visually — paste a user, tenant, and permission and see exactly which role/group
  allows it.
</Callout>

## ABAC policy

For rules RBAC can't express (e.g. attribute matching), each tenant has an ABAC
**policy** evaluated alongside RBAC. A request must satisfy both the required
permission and any matching policy.

<APIEndpoint method="GET" path="/v1/tenants/{tenantID}/policy" title="Read the tenant ABAC policy" />
<APIEndpoint method="PUT" path="/v1/tenants/{tenantID}/policy" title="Update the ABAC policy" />

<Callout type="info">
  The tenant <strong>auth policy</strong> (password rules, allowed login methods,
  breached-password rejection) is separate from this <strong>ABAC policy</strong> —
  see [Authentication → auth policy](/docs/authentication#per-tenant-auth-policy).
  Fine-grained / ReBAC (Zanzibar-style) authorization is on the roadmap.
</Callout>
