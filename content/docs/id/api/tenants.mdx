Tenants are the unit of isolation. Authenticate server-side with an API key
(`Authorization: ApiKey qk_…`).

## Lifecycle

<APIEndpoint method="POST" path="/v1/tenants" title="Create a tenant" />
<APIEndpoint method="GET" path="/v1/tenants" title="List tenants" />
<APIEndpoint method="GET" path="/v1/tenants/{id}" title="Get a tenant" />
<APIEndpoint method="PATCH" path="/v1/tenants/{id}" title="Update a tenant" />
<APIEndpoint method="DELETE" path="/v1/tenants/{id}" title="Delete a tenant" />

```bash
curl -X POST https://api.id.qeet.in/v1/tenants \
  -H "Authorization: ApiKey $QEETID_API_KEY" \
  -d '{ "name": "Acme, Inc." }'
```

## Per-tenant configuration

These are documented in detail under [Platform → Multi-tenancy](/docs/platform/multi-tenancy):

| Area | Endpoint |
| --- | --- |
| Auth policy | `GET/PUT /v1/tenants/{tenantID}/auth-policy` |
| ABAC policy | `GET/PUT /v1/tenants/{tenantID}/policy` |
| Branding | `GET/PUT /v1/tenants/{tenantID}/branding` |
| Email templates | `/v1/tenants/{tenantID}/email-templates` |
| IP rules | `/v1/tenants/{tenantID}/ip-rules` |
| Retention | `/v1/tenants/{tenantID}/retention` |
| Analytics | `GET /v1/tenants/{tenantID}/analytics/overview` |
| Audit | `GET /v1/tenants/{tenantID}/audit` (+ `/verify`) |
| Secrets | `/v1/tenants/{tenantID}/secrets` |
| SSO (SAML/LDAP/SCIM/social) | see [Enterprise](/docs/enterprise) |

<Callout type="info">
  Deleting a tenant cascades within its isolation boundary. Combine with
  [GDPR erasure](/docs/platform/gdpr) for personal-data deletion.
</Callout>
