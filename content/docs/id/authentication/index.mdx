Qeet ID supports multiple first-factor authentication methods. Which ones are
available is controlled **per tenant** by its **auth policy**, so each workspace can
allow exactly the methods it wants.

## Tokens issued on success

Every successful authentication returns a **token pair** — a short-lived **ES256
access JWT** plus a rotating **refresh token**. Verify access tokens offline against
the public JWKS; exchange refresh tokens at `/v1/auth/refresh`. See
[Sessions](/docs/sessions) for rotation, theft detection, and revocation.

<APIEndpoint method="POST" path="/v1/auth/login" title="Password login → token pair" />
<APIEndpoint method="POST" path="/v1/auth/refresh" title="Rotate the token pair" />

## Per-tenant auth policy

Each tenant has an auth policy controlling password rules, allowed login methods, and
breached-password rejection (HaveIBeenPwned k-anonymity — off by default, fail-open).

<APIEndpoint method="GET" path="/v1/tenants/{tenantID}/auth-policy" title="Read auth policy" />
<APIEndpoint method="PUT" path="/v1/tenants/{tenantID}/auth-policy" title="Update auth policy" />

<Callout type="info">
  Brute-force protection is on by default: failed attempts trigger a per-account
  throttle and temporary lockout, on top of distributed IP rate limiting. See
  [Sessions → Lockout](/docs/sessions#account-lockout) and [Rate limits](/docs/platform/rate-limits).
</Callout>

## Building the UI

You have two paths:

- **Hosted login** (`qeetid-login`) — Qeet ID hosts the full sign-in experience
  (password + passkey + social + consent). Drive it from your app with
  `@qeetid/nextjs` / `@qeetid/react`. This is the recommended path for OIDC clients.
- **Direct API** — call the REST endpoints yourself (`/v1/auth/*`, `/v1/passkeys/*`,
  `/v1/social/*`) and build your own screens.

The method pages below document the direct API; the [SDK pages](/docs/sdks) cover the
hosted flow.
