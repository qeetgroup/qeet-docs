Qeet ID ships **five first-party SDKs**. The server SDKs authenticate with an API key
(`Authorization: ApiKey qk_…`) and verify user access tokens locally against the
public **ES256 JWKS**. The web SDKs run the hosted-login OAuth flow for you.

## What the server SDKs give you

- **Local session verification** — `verify(token)` checks the ES256 signature against
  the cached JWKS, then expiry/issuer/audience. No per-request network call.
- **Single-call authorization** — `can(...)` / `canAll(...)` over `GET /v1/check`.
- **User & tenant management** — typed CRUD with cursor pagination.
- **Typed errors** + automatic retry/backoff on `429` and idempotent `5xx`
  (honoring `Retry-After`).

```ts title="the shape, across languages"
const claims = await qeetid.sessions.verify(accessToken);
const ok = await qeetid.can({ user: claims.userId, tenant: claims.tenantId!, permission: "billing:write" });
```

<Callout type="warn" title="Server SDKs use ApiKey; never ship the key">
  <code>@qeetid/sdk</code>, Go, and Python are <strong>server-side</strong> — they hold
  a <code>qk_…</code> API key. For browsers, use <code>@qeetid/nextjs</code> +
  <code>@qeetid/react</code>: the secret stays on the server and the session lives in an
  encrypted HttpOnly cookie.
</Callout>

## Planned

Node-only (CommonJS), Rust, and React Native SDKs are **planned** and not yet shipped.
Until then, use `@qeetid/sdk` (it runs on Node ≥18, Bun, Deno, and edge runtimes via
Web Crypto) or call the [REST API](/docs/api) directly.
