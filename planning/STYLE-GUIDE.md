# qeet-docs Style Guide

The conventions every page on docs.qeet.in follows. The pipeline agents
(`docs-architect`, `technical-writer`, `api-reference-engineer`,
`component-catalog-sync`, `product-accuracy-reviewer`) treat this as the contract;
`product-accuracy-reviewer` checks new content against it.

This file lives in `planning/` (an authoring aid). It is **not** published — only
`content/docs/**` is built into the site.

---

## Voice & tense

- **Second person, present tense.** "You create a key", not "The user will create a key".
- **Document what shipped.** Describe behaviour that exists in code today. Anything
  not yet built is governed by the Honesty ladder below — never blur the two.
- Lead with the reader's goal, not our architecture. Concrete over abstract.
- Short sentences. Active voice. No marketing adjectives in reference material.

## Frontmatter (required)

Every MDX page starts with:

```yaml
---
title: <Page title>              # required
description: <one search-friendly sentence>   # required — fuels search + AI chat + SEO
icon: <Lucide or custom icon name>             # required on index/section landing pages
---
```

The `description` is indexed for search and read by the "Ask AI" assistant — make it a
real, retrieval-friendly sentence, not a label.

## Diátaxis — pick the right page type

| Reader need | Page type | Template |
| --- | --- | --- |
| "Get me running fast" | Quickstart / tutorial | `templates/quickstart.template.mdx` |
| "Help me do a specific task" | How-to guide | `templates/guide.template.mdx` |
| "Help me understand the model" | Concept / explanation | `templates/concept.template.mdx` |
| "What exactly does this endpoint do" | API reference | `templates/api-page.template.mdx` |
| "What components exist" | Component catalog | `templates/component-block.template.mdx` |
| "What's this product going to be" | Building-now overview | `templates/overview-coming-soon.template.mdx` |

Keep types distinct: don't bury a tutorial inside a concept page, or vice-versa.

## Components (from `src/components/mdx.tsx`)

- `<Callout type="info|tip|warn|danger" title="…">` — status, prerequisites, warnings.
- `<Cards>` / `<Card icon title href description>` — next-steps grids and feature lists.
- `<APIEndpoint method path title />` — **only for endpoints that exist in code today.**
- Fenced code blocks always carry a language and, where useful, a `title="…"`. Use
  `<Tabs items={[…]}>` + ` ```lang tab="…" ` for multi-language samples.
- For the interactive OpenAPI reference, generated pages use `<APIPage>` — do not
  hand-author `<APIPage>`; it is emitted by `scripts/generate-openapi.mjs`.

## Honesty ladder (the rule that keeps us trustworthy)

| Product maturity | How to document it |
| --- | --- |
| **Shipped** (in code + released) | Document normally. Real `<APIEndpoint>`, real samples. |
| **Built but pre-release** (code exists, e.g. qeet-notify) | Open with `<Callout type="info" title="In active development">`. Document only what the code/OpenAPI actually exposes; mark the rest "planned". |
| **PRD-only** (no/skeleton code: people, logs, pay) | Open with `<Callout type="info" title="In design">`. Describe planned capability in prose; **never** use `<APIEndpoint>` for endpoints that don't exist. Label planned API shape "planned". |

Never document an endpoint, component, or feature that you cannot point to in source.

## Navigation (`meta.json`)

- One `meta.json` per folder. Product roots set `"root": true` (top-level tab).
- `pages` is an ordered array; group with `"---Section Title---"` string headers.
- A page listed in `pages` must exist; a new page should be added to `pages`.
- Keep `icon` consistent across like pages (e.g. all changelogs use `Package`).

## Cross-linking

- Link products with `/<product>/<page>` (e.g. `/id/quickstart`, `/notify/roadmap`).
- ⚠️ Legacy `/id` pages use bare `/docs/...` links — standardize new and edited pages
  to `/<product>/...`. (A standing cleanup item for `/id`.)

## Don'ts

- Never hand-edit `.source/` — it's regenerated by `fumadocs-mdx`.
- Never inline secrets or read `.env.local` / `*.pem` / `qeet-codes/*`.
- Gate every change on `pnpm check && pnpm typecheck && pnpm build` (run
  `nvm use v22.20.0` first).
- **Agents never commit or push.** Leave changes in the working tree for review.
