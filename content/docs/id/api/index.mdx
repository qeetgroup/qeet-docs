The Qeet ID REST API is **JSON over HTTPS**, versioned under `/v1/`. The OpenAPI spec
covers **142 paths** with **100% route coverage**, guarded in CI — it's the source of
truth for every SDK.

```text
https://api.id.qeet.in   (or your self-hosted base URL)
```

<Callout type="info" title="Self-hosted base URL">
  Qeet ID is self-hostable, so the base URL is whatever you deploy. Examples use
  <code>https://api.id.qeet.in</code>; substitute your own host. The OpenAPI
  <code>servers</code> list also includes local dev (<code>:4000</code>/<code>:4001</code>).
</Callout>

## Authentication schemes

There are several auth modes — use the right one per route:

| Scheme | Header | Used by |
| --- | --- | --- |
| **`apiKeyAuth`** | `Authorization: ApiKey qk_…` | Server-side API access. |
| **`bearerAuth`** | `Authorization: Bearer <jwt>` | User / service **ES256 JWTs**. |
| **`scimBearer`** | `Authorization: Bearer <scim-token>` | Per-tenant SCIM provisioning. |
| **`clientCredentials`** | form `client_id`/`client_secret` or HTTP Basic | OAuth M2M token/revoke/introspect. |
| **`ssoCookie`** | `qe_ls` cookie (HttpOnly) | Hosted-login SSO session flows. |

<Callout type="danger" title="ApiKey ≠ Bearer">
  The API-key scheme is literally <code>Authorization: ApiKey &lt;key&gt;</code> — not
  <code>Bearer</code>. The <code>Bearer</code> scheme on the same authenticated routes
  is for user/service JWTs. Don't send a <code>qk_…</code> key as a Bearer token.
</Callout>

```bash title="API key (server)"
curl https://api.id.qeet.in/v1/users \
  -H "Authorization: ApiKey $QEETID_API_KEY"
```

```bash title="User access token"
curl https://api.id.qeet.in/v1/auth/me \
  -H "Authorization: Bearer $ACCESS_TOKEN"
```

## Conventions

- **Tokens** are ES256 JWTs verified against `/.well-known/jwks.json`.
- **Errors** use a stable `{ "error": { "code", "message", … } }` envelope — see
  [Errors](/docs/api/errors).
- **Lists** are cursor-paginated (`items` + `next_cursor`) — see
  [Pagination](/docs/api/pagination).
- **Request IDs** are returned in the `X-Request-Id` header and echoed in error bodies.
