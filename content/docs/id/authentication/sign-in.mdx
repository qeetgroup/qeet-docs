title: Sign-in
description: Password login, the token pair it returns, and how to verify the access token against the public JWKS on your server.
---

Password sign-in exchanges an email + password for a **token pair**. There's no
publishable key and no opaque session cookie to decode — Qeet ID issues an **ES256
access JWT** your services verify offline.

## Log in

<APIEndpoint method="POST" path="/v1/auth/login" title="Password login → token pair" />

```bash title="terminal"
curl -X POST https://api.id.qeet.in/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"alex@acme.com","password":"a-strong-passphrase"}'
```

```json title="200 response — TokenPair"
{
  "access_token": "eyJhbGciOiJFUzI1Ni…",
  "token_type": "Bearer",
  "expires_at": "2026-06-04T12:15:00Z",
  "refresh_token": "…",
  "session_id": "…",
  "user_id": "…",
  "tenant_id": "…"
}
```

<Callout type="info" title="MFA-gated accounts">
  If the tenant or user requires MFA, complete the appropriate factor before the
  session is fully privileged. See [MFA](/docs/mfa) for the factor and step-up flows.
</Callout>

## Verify the access token on your server

Send the access token to your backend as a bearer header, then verify it locally
against the cached JWKS — signature, expiry, issuer, audience. No network call per
request.

<Tabs items={['TypeScript', 'Go', 'Python']}>

```ts tab="TypeScript"
import { qeetid } from "@/lib/qeetid";

const claims = await qeetid.sessions.verify(accessToken);
// claims.userId, claims.tenantId, claims.sessionId
```

```go tab="Go"
claims, err := qeetid.Sessions.Verify(ctx, accessToken)
```

```python tab="Python"
claims = qeetid.sessions.verify(access_token)
```

</Tabs>

<Callout type="warn" title="Bearer = the JWT, not the API key">
  User access tokens use <code>Authorization: Bearer &lt;jwt&gt;</code>. Server-side
  API keys use <code>Authorization: ApiKey qk_…</code>. They are different schemes —
  don't put an API key behind <code>Bearer</code> or vice versa.
</Callout>

## Failure modes

| Failure | HTTP | Notes |
| --- | --- | --- |
| Wrong email/password | `401` | Generic — never reveal which field. |
| Too many attempts | `429` | Per-account lockout + IP rate limit kicked in. See [Sessions → Lockout](/docs/sessions#account-lockout). |
| Validation error | `422` | Malformed body. |

The error body always follows the `{ "error": { "code", "message" } }` envelope —
see [API → Errors](/docs/api/errors).

## Hosted login

To avoid building login UI at all, run the OAuth Authorization Code + PKCE flow
against Qeet ID's hosted login with `@qeetid/nextjs`. The SDK handles the token pair,
stores an encrypted HttpOnly cookie, and refreshes silently. See
[SDKs → TypeScript & Next.js](/docs/sdks/typescript).
