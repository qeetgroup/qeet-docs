Qeet ID ships a complete multi-factor system: **TOTP** with recovery codes,
**email/SMS OTP factors**, **WebAuthn as a second factor**, and **step-up MFA** that
gates sensitive actions behind a recent verification.

<Callout type="tip" title="All implemented">
  TOTP, OTP factors, WebAuthn-2FA, and step-up are live. Any factor verification
  refreshes a recent-MFA window that a server-side <code>RequireRecentMFA</code>
  middleware checks before sensitive operations.
</Callout>

## Factors

| Factor | Endpoints | Notes |
| --- | --- | --- |
| **TOTP** (RFC 6238) | `/v1/mfa/totp/*` | 6-digit authenticator codes; enroll → confirm → verify. |
| **Recovery codes** | `/v1/mfa/recovery-codes*` | bcrypt-hashed single-use codes; regenerable. |
| **Email / SMS OTP** | `/v1/mfa/otp/factors*` | Per-factor challenge + confirm via real senders. |
| **WebAuthn 2FA** | `/v1/mfa/webauthn/*` | Reuses the user's registered passkeys as a second factor. |

## TOTP

Three-step enrollment, then verification at sign-in.

<Steps>

<Step>
### Start enrollment

Returns the shared secret + provisioning URI to render as a QR code.

<APIEndpoint method="POST" path="/v1/mfa/totp/enroll/start" title="Begin TOTP enrollment" />
</Step>

<Step>
### Confirm

The user enters the first 6-digit code to prove the authenticator is synced.

<APIEndpoint method="POST" path="/v1/mfa/totp/enroll/confirm" title="Confirm enrollment" />
</Step>

<Step>
### Verify at sign-in

<APIEndpoint method="POST" path="/v1/mfa/totp/verify" title="Verify a TOTP code" />
</Step>

</Steps>

Remove TOTP with `DELETE /v1/mfa/totp` (this is a sensitive action — see
[Step-up](#step-up-mfa)).

## Recovery codes

Issued at enrollment, single-use, bcrypt-hashed. Users can regenerate the set
(another step-up-gated action).

<APIEndpoint method="GET" path="/v1/mfa/recovery-codes" title="List remaining recovery codes" />
<APIEndpoint method="POST" path="/v1/mfa/recovery-codes/regenerate" title="Regenerate the set" />

## Email / SMS OTP factors

Register a contact factor, then challenge + confirm it as a second step.

<APIEndpoint method="POST" path="/v1/mfa/otp/factors" title="Register an OTP factor (email/SMS)" />
<APIEndpoint method="POST" path="/v1/mfa/otp/factors/{id}/challenge" title="Send a code" />
<APIEndpoint method="POST" path="/v1/mfa/otp/factors/{id}/confirm" title="Confirm enrollment" />
<APIEndpoint method="POST" path="/v1/mfa/otp/verify" title="Verify an OTP at sign-in" />

<Callout type="warn" title="SMS is lower assurance">
  Prefer TOTP or WebAuthn as the primary second factor; SMS is vulnerable to SIM-swap.
  SMS/email delivery also depends on a configured sending domain / Twilio account.
</Callout>

## WebAuthn as a second factor

Step-up with a registered passkey. Unlike passwordless login, this **asserts the
known user** and issues no new token — it only proves presence to refresh the MFA
window.

<APIEndpoint method="POST" path="/v1/mfa/webauthn/challenge" title="Get assertion options" />
<APIEndpoint method="POST" path="/v1/mfa/webauthn/verify" title="Verify the assertion" />

## Step-up MFA

Any successful factor verification refreshes a **recent-verification window**. The
`RequireRecentMFA` middleware gates sensitive endpoints (e.g. disabling MFA,
regenerating recovery codes) and returns an MFA challenge if the window has lapsed.

Check whether the user currently satisfies the window:

<APIEndpoint method="GET" path="/v1/mfa/step-up/status" title="Is a recent MFA present?" />

```bash
curl https://api.id.qeet.in/v1/mfa/step-up/status \
  -H "Authorization: Bearer $ACCESS_TOKEN"
# → { "recent": false }  →  prompt the user to re-verify a factor
```

<Callout type="info">
  The pattern: call <code>step-up/status</code> before a sensitive action; if not
  recent, run any factor's verify endpoint to refresh the window, then retry. The
  server enforces this independently — clients can't bypass it.
</Callout>
