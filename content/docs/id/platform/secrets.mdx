Qeet ID includes a per-tenant **secrets vault**: store sensitive values (provider
credentials, signing material, integration secrets) encrypted with **AES-256-GCM**.
The encryption key is sourced through a **`KeyProvider`** abstraction — decoupled from
the JWT secret — so the underlying key management is swappable.

## Store & retrieve

<APIEndpoint
  method="GET"
  path="/v1/tenants/{tenantID}/secrets"
  title="List secrets (metadata only)"
/>
<APIEndpoint
  method="POST"
  path="/v1/tenants/{tenantID}/secrets"
  title="Create a secret"
/>
<APIEndpoint
  method="DELETE"
  path="/v1/tenants/{tenantID}/secrets/{id}"
  title="Delete a secret"
/>
<APIEndpoint
  method="POST"
  path="/v1/tenants/{tenantID}/secrets/{id}/reveal"
  title="Reveal a secret value"
/>

Listing returns metadata only; the plaintext value is returned solely by the explicit
**reveal** endpoint, which is audited.

<Callout type="warn" title="Reveal is sensitive">
  Every reveal is recorded in the [audit log](/id/platform/audit). Treat reveal
  as a break-glass operation, not a routine read path.
</Callout>

## KeyProvider & BYOK

The vault key comes from a `KeyProvider` interface. The default is local key material;
an **AWS KMS** provider drops into the same interface for **BYOK / envelope
encryption** — you supply a real KMS key. The same abstraction backs JWT/OIDC signing
key material.

<Callout type="info">
  KMS-backed keys are an external dependency (you provide the cloud key), not a
  code gap — the interface is in place and the AWS KMS provider plugs in. See
  the deployment [runbook](/id/platform/edge#deploying-qeet-id).
</Callout>
